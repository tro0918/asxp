Agent Skill Exchange Protocol · Draft 0.1

에이전트의 능력을
파일 하나로 교환한다

ASXP는 에이전트의 재사용 가능한 prompt와 tool 계약을 skill.md로 내보내고, 다른 에이전트가 인라인으로 가져오며, 필요할 때만 원본 에이전트에 위임하도록 정의한 벤더 중립 프로토콜 초안입니다.

Problem

도구 호출과 능력 이전은 다릅니다

MCP가 에이전트가 외부 도구를 호출하는 공통 인터페이스라면, ASXP는 한 에이전트의 정의와 실행 능력 자체를 다른 에이전트가 가져오는 교환 계층입니다.

01

정의의 이식

재사용 가능한 prompt와 도구 입력·출력 계약을 하나의 읽기 쉬운 문서로 이동합니다.

02

선택적 연결

정적 파일로 끝낼 수도 있고, 원본 에이전트와 연결해 최신 버전을 다시 동기화할 수도 있습니다.

03

지식의 격리

RAG, 지식 파일, 자격증명은 이동하지 않습니다. 필요할 때만 원본에 최소 범위로 질의합니다.

한 문장 정의
ASXP는 “prompt + tool contract는 복사하고, private knowledge + credential은 원본에 남기는” 에이전트 스킬 교환 프로토콜입니다.
Core model

하나의 파일, 두 가지 실행 방식

INLINE

가져온 능력을 로컬에서 실행

  • Markdown body를 스킬 지침으로 주입
  • ae-tools-json 도구를 부모의 tools 배열에 평탄화
  • 호스트 정책과 사용자 권한 안에서 실행
  • 원본이 없어도 정적 snapshot으로 동작
DELEGATE · OPTIONAL

원본 지식이 필요할 때만 위임

  • 연결된 스킬에 ask_<safeName> 합성
  • 원본 RAG나 비공개 데이터는 이동하지 않음
  • 최소 task context만 전송
  • OAuth scope, consent, depth limit 적용
Lifecycle

Export에서 Re-sync까지

STEP 1

Export

에이전트의 재사용 가능한 prompt와 공개 tool contract를 skill.md로 생성합니다.

→
STEP 2

Validate

필수 필드, JSON Schema, URL, 권한, credential 포함 여부를 검사합니다.

→
STEP 3

Import

파일 또는 sibling agent에서 가져오고, 충돌 도구는 안전하게 namespace 처리합니다.

→
STEP 4

Run / Sync

로컬에서 실행하고, origin이 있으면 immutable revision 기준으로 다시 동기화합니다.

가장 작은 호환 문서

---
name: customer-support
description: 주문 관련 질문을 처리할 때 사용하는 고객지원 스킬.
asxp-version: 0.1
version: 1.0.0
permissions: orders.read
privacy: local
ae-tools-json: '[{"name":"lookup_order","description":"주문 조회","inputSchema":{"type":"object","properties":{"order_id":{"type":"string"}},"required":["order_id"]}}]'
---

# Customer support
주문 번호가 없으면 먼저 요청하고, 확인되지 않은 배송일은 추측하지 마세요.
Runtime

실행 중에는 이렇게 결합됩니다

Instructions
활성화된 skill body가 ## Available skills 아래의 격리된 지침 영역에 추가됩니다. 호스트 정책보다 높은 권한을 갖지는 않습니다.
Tools
ae-tools-json의 도구들이 부모 runtime에 노출됩니다. 이름 충돌 시 <skill>__<tool>처럼 결정적으로 namespace 처리합니다.
Delegation
origin이 살아 있는 경우에만 로컬 합성 도구를 제공합니다. 이 도구는 exported tool이 아니라 연결 상태에서 만들어지는 runtime binding입니다.
Guardrails
idempotency key, trace ID, visited set, recursion depth, request size, privacy scope를 모든 위임 경로에 적용합니다.
Conformance

처음부터 모든 기능이 필요하지 않습니다

구현체는 필요한 수준까지만 지원하고 자신이 지원하는 프로필을 선언할 수 있습니다.

Security boundary

무엇을 옮기고, 무엇을 남길 것인가

포함 가능

  • 재사용 가능한 agent instruction
  • 도구 이름과 설명
  • JSON Schema 기반 입력·출력 계약
  • 필요 권한과 privacy 선언
  • origin URL, revision, publisher identity
  • 서명 검증용 key identifier

포함 금지

  • API key, OAuth token, cookie, certificate
  • Slack·LINE 등 채널 credential
  • 원본 RAG 파일, embedding, private index
  • 대화 기록과 사용자 개인정보
  • 숨겨진 플랫폼 정책
  • 내부 전용 endpoint와 구현 secret
서명은 안전성 인증서가 아닙니다.
JWS 서명은 발행자 키 보유와 파일 무결성을 증명합니다. 스킬이 유해하지 않다는 의미는 아니므로 importer의 권한 검토와 로컬 정책 적용이 반드시 필요합니다.
Adoption

현실적인 도입 순서

  1. Core부터 배포
    현재 Skills 탭의 Download / Upload에 parser와 validator를 연결합니다.
  2. Sibling import를 Linked로 전환
    sourceAgentId를 외부 표현인 origin URI와 immutable revision으로 매핑합니다.
  3. 권한 차이를 보여주는 Re-sync
    새 tool이나 privacy 범위가 추가되면 자동 활성화하지 않고 사용자 승인을 받습니다.
  4. Delegation을 선택적으로 개방
    private knowledge가 필요한 use case부터 OAuth scope와 consent를 적용합니다.
  5. Registry와 서명 도입
    파트너 또는 공개 생태계로 확장할 때 publish, resolve, revoke, JWKS를 운영합니다.
